## Architecture Overview

All admin pages are encrypted with [StatiCrypt](https://github.com/robinmoisson/staticrypt). The dashboard (`index.html`) loads other tools in an iframe. A single login unlocks every tab because every file shares the same salt and localStorage keys.

**Design principles:**
- Every tool is a self-contained single HTML file with inline CSS/JS
- No build step for admin pages - encrypt and commit
- All data loaded from static JSON files generated by Python scripts during Jekyll build
- Direct GitHub API writes (via PAT) for tools that need to commit changes back to the repo
- Single shared salt across all pages means single sign-on behavior

**Why a PAT is embedded in the client:** The admin pages are encrypted with a 4-character password (`7777`). Anyone who unlocks the dashboard has full write access to the repo. The PAT is a convenience, not a security boundary. The actual security comes from the StatiCrypt encryption layer.

## Encryption System

### Critical: Shared Salt

All encrypted files MUST use the same salt for single sign-on to work:

```
0225b0a7b3aac02f31e146d7af8711cb
```

If files are encrypted with different salts, each iframe tab prompts for the password individually.

### localStorage Keys (shared across all pages)

```
rememberExpirationKey: "staticrypt_expiration"
rememberPassphraseKey: "staticrypt_passphrase"
rememberDurationInDays: 30
```

### PBKDF2 Decrypt Process (3 rounds)

StatiCrypt uses a 3-round key derivation:

1. **Round 1:** PBKDF2(password, salt, 1000 iterations, SHA-1)
2. **Round 2:** PBKDF2(round1_hex, salt, 14000 iterations, SHA-256)
3. **Round 3:** PBKDF2(round2_hex, salt, 585000 iterations, SHA-256)

Total: 600,000 iterations. Salt is UTF-8 encoded (NOT hex-decoded) when passed to PBKDF2.

**Signed message format:**
- First 64 hex chars = HMAC
- Rest = encrypted payload

**Encrypted payload format:**
- First 32 hex chars = IV (16 bytes)
- Rest = AES-CBC ciphertext

### Common Mistakes to Avoid

- Using a different salt (breaks single sign-on)
- Forgetting `--short` flag (StaticCrypt rejects 4-char password interactively)
- Forgetting `-c false` (creates `.staticrypt.json` that shouldn't be committed)
- Encrypting without using the custom password template (breaks visual consistency)

## Fast-Track Workflow for Claude

When asked to modify or add admin pages, follow this exact sequence to avoid discovery overhead.

### One-Command Decrypt

```bash
cd admin
npx staticrypt --decrypt -p 7777 -s 0225b0a7b3aac02f31e146d7af8711cb --short -d decrypted -c false TARGET.html
```

Replace `TARGET.html` with one or more filenames. Decrypted inner HTML lands in `admin/decrypted/TARGET.html`.

### One-Command Encrypt

```bash
npx staticrypt INNER.html \
  -p 7777 \
  -s 0225b0a7b3aac02f31e146d7af8711cb \
  -t admin/ic-password-template.html \
  -d output \
  --remember 30 \
  --template-title "Admin Area" \
  --template-button UNLOCK \
  --template-placeholder "Enter password" \
  --template-error "Incorrect password. Please try again." \
  --template-instructions "Enter the admin password to access this page." \
  --short \
  -c false
```

Multiple files can be passed in one call. Output lands in `output/INNER.html`.

### Verification (mandatory before delivery)

```bash
# 1. Salt present
grep -c '0225b0a7b3aac02f31e146d7af8711cb' output/TARGET.html
# Must print 1

# 2. Round-trip decrypt
npx staticrypt --decrypt -p 7777 -s 0225b0a7b3aac02f31e146d7af8711cb --short -d verify -c false output/TARGET.html
head -1 verify/TARGET.html
# Must show <!DOCTYPE html>

# 3. Confirm edits are in decrypted output (not just encryption wrapper)
grep -c "YOUR_CHANGE_MARKER" verify/TARGET.html
# Must be > 0
```

### Step-by-Step: Modify an Existing Page

1. Install staticrypt once per session: `npm install -g staticrypt`
2. Decrypt the target: `npx staticrypt --decrypt ... TARGET.html`
3. Apply edits to `decrypted/TARGET.html`
4. Re-encrypt: `npx staticrypt decrypted/TARGET.html ...`
5. Verify salt + round-trip
6. Rename/copy `output/TARGET.html` to final delivery name
7. Deliver

### Step-by-Step: Add a New Page

1. Build the inner HTML as a standalone file (no encryption wrapper, no staticrypt references)
2. Decrypt `index.html` (the dashboard)
3. In decrypted `index.html`, add:
   - A sidebar `<li>` with `data-page="newpage.html"` and an SVG icon under the appropriate `sidebar-section`
   - A welcome card `<a class="wcard">` in the `welcome-grid` div
   - An optional `.wcard-icon.newkey` class in the CSS for the card icon color
4. Encrypt BOTH the new page AND the updated `index.html` with the shared salt
5. Verify salt + round-trip for both
6. Deliver both encrypted files

### Flags Reference

| Flag | Why |
|------|-----|
| `--short` | Suppresses the "password too short" interactive prompt (our password is 4 chars) |
| `-c false` | Disables `.staticrypt.json` config file creation |
| `-s SALT` | Always pass the shared salt explicitly |
| `--remember 30` | Sets localStorage remember-me to 30 days |
| `-t TEMPLATE` | Points to the custom ic-password-template.html for consistent styling |
| `-d OUTPUT_DIR` | Output directory for encrypted files |

## Tool Reference

### Create Post (`create-post.html`)

Rich markdown editor with live preview, validation, and direct GitHub push.

**Features:**
- Split-pane layout: frontmatter form (top), markdown editor + preview (bottom)
- Auto-slug generation from title with manual override
- Auto reading_time calculation (words ÷ 238)
- Image rotation formula: `((N-1) % 15) + 1` with auto-preview
- Author dropdown loaded from `authors-data.json`
- Language dropdown: en, bn, zh, hi, ta, te, kn, ml, ur
- 8 series templates: SAT, UPSC, World History, Harry Potter, Feluda EN, Feluda BN, Gaokao, Generic
- Keyboard shortcuts: Cmd+B (bold), Cmd+I (italic), Cmd+K (link), Cmd+S (save draft)
- Draft autosave to localStorage every 10 seconds
- Quality checks panel: title/slug/date/excerpt/image/author (critical), word count ≥ 14,000, em dash detection, bullet point detection, FAQ count (20 expected), word repeat limit (≤ 50 non-stop-word repeats)
- Push button disabled until all critical checks pass
- Confirm modal showing filename, title, author, word count before push
- Commit message includes `[skip ci]` so posts wait for next cron build (up to 4 days)

**Push mechanism:** Uses GitHub Git Data API (ref &rarr; blob &rarr; tree &rarr; commit &rarr; ref update). Same pattern as `word-frequency.html` `commitNow()`. See [GitHub API Push Pattern](#github-api-push-pattern).

**Localstorage key:** `ic_createpost_draft`

### Posts Dashboard (`posts.html`)

Table of every published post with search, filters, and SEO scores.

**Loads from:** `posts-data.json`

**Checks performed inline:** SEO score calculation from title length, excerpt length, word count, tags, image presence.

### Scheduled Posts (`scheduled.html`)

Lists future-dated posts not yet published.

**Data source:** Public GitHub API. Fetches `_posts/` file list, filters by future dates in filenames, then fetches raw markdown from `raw.githubusercontent.com` to parse YAML frontmatter.

**Why GitHub API instead of posts-data.json:** Jekyll `_config.yml` has `future: false`, which excludes future posts from all Liquid loops including `posts-data.json`. No API key required since repo is public.

### Posts Doctor (`doctor.html`)

Quality diagnostics across every post. Shows only posts with issues.

**Loads from:** `posts-data.json`

**Issue types:**
- Missing featured image
- Missing excerpt
- No tags
- Thin content (&lt; 300 words)
- Short content (&lt; 600 words)
- Title too long/short
- Slug too long
- Poor SEO score (&lt; 50)

Classified as **critical** or **warning**. Filters allow drilling by issue type, category, and sort order.

### Article Ideas (`article-ideas.html`)

AI-powered topic discovery from RSS signals. Coverage gaps, heat scores, competition analysis.

### Link Inspector (`link-inspector.html`)

**Loads from:** `related-data.json` (from `_data/related_posts.yml`) and `posts-data.json`

**Six tabs:**
- **Overview** - histogram, top/bottom linked posts
- **Inbound Links** - searchable/filterable table
- **Outbound Links** - with cross-category count
- **Orphan Check** - zero-inbound detection
- **Category Clusters** - intra vs inter linking stats
- **Weakest Links** - posts with fewest total connections

Requires `_scripts/generate_related_posts.py` to populate `_data/related_posts.yml` first.

### Keyword Audit (`keyword-audit.html`)

**Loads from:** `keyword-audit-data.json`

Detects posts where proper nouns (cities, companies, technologies, exams, programs) in URL slug or title are missing from `page_title` and/or post body. Catches SEO gaps where a user clicks through expecting content about a topic (e.g., "TCS ILP Bhubaneswar") but the body never mentions that keyword.

**Issue types:**
- `SLUG_KEYWORD_MISSING_FROM_BODY` (critical) - proper noun in slug, zero body mentions
- `SLUG_KEYWORD_SPARSE_IN_BODY` (warning) - fewer than 3 mentions in 2000+ word post
- `TITLE_KEYWORD_MISSING_FROM_BODY` (critical)
- `TITLE_KEYWORD_SPARSE_IN_BODY` (warning)
- `MISSING_FROM_PAGE_TITLE` (critical)
- `NO_PAGE_TITLE` (warning) - post lacks `page_title` field entirely

The known-noun dictionary in `_scripts/keyword_audit.py` can be expanded as new proper nouns become relevant.

### Word Frequency (`word-frequency.html`)

**Loads from:** `word-frequency-data.json`

Detects filler-word repetition. Any non-stop-word appearing more than 50 times triggers a flag. Canonical failure: "specific" appearing 2,237 times in one article.

**Workflow:**
- Posts flagged automatically during scheduled cron builds
- Admin reviews each flag and either fixes the post or marks it OK (legitimate topic word, character name, etc.)
- "Mark OK" adds a `word_frequency_ok` reason to the post's frontmatter
- Queued Mark OK entries auto-commit after 90 seconds idle, or via "Commit now" button
- Commit message: `admin: mark N posts as word-frequency OK [skip ci]`

**Why `[skip ci]`:** The Mark OK action only changes metadata that doesn't affect rendered output. Skipping the build saves CI minutes. The next scheduled cron picks up the changes.

### Broken Links (`broken-links.html`)

**Loads from:** `links-data.json`

External link health monitor. Flags 4xx, 5xx, and timeout responses. Cache stores per-URL checked dates with 14-day expiry to avoid hammering destination sites.

### Author Map (`author-map.html`)

**Loads from:** `authors-data.json` and a voice quote bank

Visualizes the 121-author pool across 18 pools. Daily rotating voice quotes per author seeded by age bracket, morale tier, and deterministic RNG. Click any author to see profile, credential, pool, age, post count, morale emoji, bio.

Authors assigned to posts via `_data/author_rules.yml` rule ordering (`western-tech` rules must come before `western-edu` to prevent ACT/WARN Act collision).

### Content Pipeline (`pipeline.html`)

Hub-spoke dashboard for series planning. See [Hub-Spoke Content Plan](#hub-spoke-content-plan).

## Data Pipeline

### End-to-End Flow

```
_posts/*.md (markdown files)
    ↓
Jekyll build (runs on push + cron)
    ↓ ↓ ↓ ↓ ↓
    ├─ posts-data.json (Liquid template)
    ├─ related-data.json (Liquid template)
    ├─ authors-data.json (Liquid template)
    ├─ keyword-audit-data.json (Python script)
    ├─ word-frequency-data.json (Python script)
    └─ links-data.json (Python script)
    ↓
admin/*.html tools load these JSON files
    ↓
admin actions may push changes back to _posts/ via GitHub API
    ↓ (with [skip ci])
    (committed but no rebuild)
    ↓
Next cron build picks up changes, regenerates data files, rebuilds site
```

### Why data files are committed (not gitignored)

- Admin tools load them at runtime from the live site
- They are build outputs, treated as intermediate artifacts
- Committing them means any developer can browse admin tools without running the full pipeline locally
- The scheduled cron regenerates them fresh every 4 days

### `posts-data.json` Performance Warning

Avoid using `{{ post.content | size }}` or any filter that forces full markdown-to-HTML rendering of every post's body. With 1,300+ posts, this hits GitHub Pages' Liquid processing limits and silently truncates JSON output.

**Safe fields:** `number_of_words`, `title | size`, `excerpt | strip_html | size`, frontmatter fields.

**Unsafe fields:** `content`, `content | size`, anything that invokes the Kramdown renderer.

## GitHub Actions Workflow

Single workflow: `.github/workflows/jekyll.yml`

### Triggers

| Trigger | Frequency | Runs Scripts |
|---------|-----------|--------------|
| `push` to `main` | Immediate | Image assignment, author assignment, language tagging |
| `schedule` cron | Every 4 days, 6 AM UTC | Freshness rotation, keyword audit, word frequency audit, link extraction, related posts generation |
| `workflow_dispatch` | Manual | Same as schedule |

### Step-by-step flow

**On push event:**
1. Checkout (with write token)
2. Setup Ruby + Bundler cache
3. Setup Pages
4. Setup Python + pyyaml
5. **Run push-only scripts:**
   - `assign_blog_images.py` - ensures new post has an image
   - `assign_authors.py` - applies author rules
   - `add_lang_field.py` - detects and tags language
6. Commit automated changes (with `[skip ci]`)
7. Build with Jekyll
8. Upload artifact
9. Deploy to Pages

**On schedule/manual event:**
1-4. Same setup as push
5. **Run schedule-only scripts:**
   - `update_freshness.py --batch-size 15` - rotates last_updated on 15 oldest posts
   - `keyword_audit.py` - regenerates keyword-audit-data.json
   - `word_frequency_audit.py` - regenerates word-frequency-data.json
   - `extract_links.py` - refreshes link health cache
   - `generate_related_posts.py` - rebuilds _data/related_posts.yml
6. Commit automated changes (with `[skip ci]`)
7-9. Build, upload, deploy (same as push)

### Skipping Builds

Commits containing `[skip ci]` in the message body skip the push trigger. This is used:
- By the workflow itself when auto-committing generated data files (prevents infinite loop)
- By admin tools (`create-post.html`, `word-frequency.html` Mark OK) to defer the build to the next cron run

**What users should know:** If you push with `[skip ci]`, the post/change is in the repo but the live site does NOT rebuild until the next cron runs (up to 4 days later). This is intentional for batch operations and draft-style workflows.

### Why `future: false` matters

`_config.yml` has `future: false`, which excludes future-dated posts from all Jekyll `site.posts` loops. This affects:
- `posts-data.json` (excludes future posts)
- Post indices and category pages
- Scheduled posts tool uses GitHub API directly because of this

## GitHub API Push Pattern

Pattern used by `word-frequency.html` and `create-post.html` to write files directly to the repo without going through the GitHub web UI.

### Embedded PAT

```javascript
var GH_TOKEN='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';
var GH_REPO='insightcrunch/insightcrunch';
var GH_BRANCH='main';
```

The PAT has write access to the repo. It's embedded in the encrypted HTML, so it's only accessible after StatiCrypt decryption.

### Git Data API sequence (5 calls)

```javascript
var headers={'Authorization':'token '+GH_TOKEN,'Accept':'application/vnd.github.v3+json'};
var apiBase='https://api.github.com/repos/'+GH_REPO;

// 1. Get current ref (HEAD of branch)
var refResp = await fetch(apiBase+'/git/ref/heads/'+GH_BRANCH,{headers});
var ref = await refResp.json();
var parentSha = ref.object.sha;

// 2. Get parent commit (need tree SHA)
var commitResp = await fetch(apiBase+'/git/commits/'+parentSha,{headers});
var parentCommit = await commitResp.json();
var baseTreeSha = parentCommit.tree.sha;

// 3. Create blob for each file
var blobResp = await fetch(apiBase+'/git/blobs',{
  method:'POST', headers,
  body:JSON.stringify({
    content: btoa(unescape(encodeURIComponent(fileContent))),
    encoding: 'base64'
  })
});
var blob = await blobResp.json();

// 4. Create tree
var treeResp = await fetch(apiBase+'/git/trees',{
  method:'POST', headers,
  body:JSON.stringify({
    base_tree: baseTreeSha,
    tree: [{path: '_posts/FILENAME.md', mode: '100644', type: 'blob', sha: blob.sha}]
  })
});
var newTree = await treeResp.json();

// 5. Create commit
var commitNewResp = await fetch(apiBase+'/git/commits',{
  method:'POST', headers,
  body:JSON.stringify({
    message: 'commit message [skip ci]',
    tree: newTree.sha,
    parents: [parentSha]
  })
});
var newCommit = await commitNewResp.json();

// 6. Update ref (PATCH, not POST)
await fetch(apiBase+'/git/refs/heads/'+GH_BRANCH,{
  method:'PATCH', headers,
  body:JSON.stringify({sha: newCommit.sha})
});
```

Total: ~2 seconds for a single-file commit. Much faster than GitHub web UI.

### When to use `[skip ci]` vs trigger build

| Scenario | Use `[skip ci]`? | Reason |
|----------|-----------------|--------|
| Adding a new post | Yes (current policy) | Batch with next cron, prioritize writing UX |
| Editing existing post | Yes | Same as above |
| Metadata-only change (Mark OK, etc.) | Yes | No visible change to rendered output |
| Urgent typo fix | No | Need immediate deploy |

### Notes on `encoding: 'base64'`

File content must be base64-encoded when creating a blob. The idiomatic UTF-8 safe encoding is `btoa(unescape(encodeURIComponent(content)))`. This handles Unicode characters (Bengali, Chinese, etc.) correctly.

### Handling existing files (overwrites)

Check existence first:

```javascript
var exists = await fetch(apiBase+'/contents/'+encodeURIComponent(path)+'?ref='+GH_BRANCH, {headers});
if (exists.ok) {
  if (!confirm('File exists. Overwrite?')) return;
}
```

Overwrite is supported natively by creating a new tree with the same path - Git creates a new commit on top of the old blob.

### Ref drift safety

If another commit landed while the user was editing, the PATCH to `/git/refs/heads/main` may succeed but create a non-fast-forward commit. For production use, consider:
- Fetching the ref again immediately before the final PATCH
- Rebasing the new blob onto the newer tree
- Detecting ref drift and warning the user

Current tools accept last-write-wins for simplicity.

## Python Build Scripts

Location: `_scripts/`

| Script | Purpose | Trigger |
|--------|---------|---------|
| `add_lang_field.py` | Detects writing system (Latin/CJK/Bengali/etc.) per post and adds `lang:` frontmatter field | push only |
| `assign_authors.py` | Applies `_data/author_rules.yml` to assign each post an author slug, writes `_data/post_authors.yml` | push only |
| `assign_blog_images.py` | Distributes `assets/images/blog/*.webp` across posts using hash-based assignment. Replaces placeholders and fills missing `image:` frontmatter | push only |
| `extract_links.py` | Scans all markdown posts, extracts links, checks internal against post slugs, checks external with concurrent HEAD requests. 14-day cache. Outputs `admin/links-data.json` | schedule only |
| `generate_related_posts.py` | Computes pairwise post similarity (category + tags with partial matching + title tokens + first-100-word overlap), writes `_data/related_posts.yml` | schedule only |
| `keyword_audit.py` | Known-noun dictionary scanner for slug/title vs body gaps. Outputs `admin/keyword-audit-data.json` | schedule only |
| `update_freshness.py --batch-size 15` | Rotates `last_updated` on 15 oldest posts, rotates meta description from `meta_variants.json` | schedule only |
| `word_frequency_audit.py` | Counts non-stop-word frequency across all posts, flags any exceeding 50 occurrences. Outputs `admin/word-frequency-data.json` | schedule only |

### Author Rules Ordering

`_data/author_rules.yml` is evaluated top-to-bottom. **Order matters.** Known conflicts:

- `western-tech` rules must come before `western-edu` rules to prevent ACT/WARN Act keyword collision
- General category rules should come before specific tag rules if tags are ambiguous

### Unicode Normalization

`generate_related_posts.py` applies Unicode NFC normalization to Bengali slugs to handle NFD-decomposed filenames consistently. Without this, the same conceptual Bengali character could produce different slug strings depending on source.

### Adding a new `wcard-icon` color

In `index.html` CSS, add a class matching the convention:

```css
.wcard-icon.yourkey{background:rgba(R,G,B,.12);color:#HEX}
```

Then use `<div class="wcard-icon yourkey">&#ICON_ENTITY;</div>` in the card.

### Toast Pattern

```javascript
function toast(msg, type){
  var t = document.getElementById('toast');
  t.textContent = msg;
  t.className = 'toast show ' + (type || '');
  setTimeout(function(){t.className = 'toast ' + (type || '')}, 4500);
}
```

Types: `success`, `error`, `info`.

## Troubleshooting

### "Password prompt appears twice"

Cause: Different salts used during encryption.
Fix: Re-encrypt all files with the shared salt.

### "Page shows raw HTML after login"

Cause: Missing `<!DOCTYPE html>` as first line of the inner (decrypted) file, or the file was double-encrypted.
Fix: Decrypt the problem file, verify it starts with `<!DOCTYPE html>`, then re-encrypt.

### "Changes don't appear after encrypting"

Cause: Forgot to re-encrypt after editing, or encrypted the wrong file.
Fix: Run round-trip verification on the encrypted output: `head -5 verify/TARGET.html` must show the real inner HTML, not the staticrypt wrapper.

### "Tool loads but data is stale"

Cause: JSON data file in repo is older than the admin tool expects.
Fix: Trigger a manual `workflow_dispatch` run on the Jekyll workflow to regenerate all data files.

### "Create post push fails with 401"

Cause: PAT expired or was rotated.
Fix: Generate new PAT with `contents:write` scope for `insightcrunch/insightcrunch`, update `GH_TOKEN` in both `word-frequency.html` AND `create-post.html` decrypted sources, re-encrypt both.

### "Commit succeeded but post not live after cron"

Causes to check in order:
1. Commit message had `[skip ci]` - this is expected, wait for next cron (up to 4 days)
2. Date in filename is future-dated - `future: false` excludes it from build
3. Frontmatter has `published: false` - remove the field or set to true
4. Slug conflicts with existing post - Jekyll picks one, drops the other silently

### "Liquid template renders partially / silently truncates"

Cause: Used `post.content` in a loop over `site.posts` in `posts-data.json` or similar template.
Fix: Remove content-rendering filters. Use only lightweight metadata fields.

### "Author dialog shows wrong author for post"

Cause: `_data/post_authors.yml` out of sync with frontmatter.
Fix: Run `assign_authors.py` locally or trigger a manual workflow run.

### "Word frequency Mark OK button doesn't commit"

Cause: Queue empty or PAT issue.
Fix: Check browser console. Common issues: rate limit, invalid PAT, merge conflict from rapid sequential commits.

# Decrypt one or more admin files
npx staticrypt --decrypt -p 7777 -s 0225b0a7b3aac02f31e146d7af8711cb --short -d decrypted -c false FILE.html

# Encrypt one or more inner HTML files
npx staticrypt FILE.html -p 7777 -s 0225b0a7b3aac02f31e146d7af8711cb -t admin/ic-password-template.html -d output --remember 30 --template-title "Admin Area" --template-button UNLOCK --template-placeholder "Enter password" --template-error "Incorrect password. Please try again." --template-instructions "Enter the admin password to access this page." --short -c false

# Round-trip verify
grep -c '0225b0a7b3aac02f31e146d7af8711cb' output/FILE.html    # must be 1

head -1 verify/FILE.html    # must be <!DOCTYPE html>

### Trigger Manual Build

```
GitHub web UI -> Actions tab -> "Deploy Jekyll site to Pages" workflow -> "Run workflow" button (runs all scripts + build)
```

Useful when:
- Testing scheduled-only scripts locally isn't feasible
- A cron run was missed or failed
- After rotating `POOL_HUBS` or updating author rules
- After bulk pushing posts with `[skip ci]` and wanting them live immediately
